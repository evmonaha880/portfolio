The systems I run, and what broke along the way

I run my work on a fleet of 29 scheduled agents, built with Claude Code, on a Mac mini in my apartment. On the morning of July 6 the fleet ran 32 jobs with zero false alarms. That was one morning's run. Some mornings a job dies, and the ticket printer says so. This page is the part of that work I can show.

The operating system

Twenty-nine scheduled agents and a heartbeat

Laika is the system that runs my day. This week's schedule holds 29 agents; a thirtieth is installed but switched off. The roster: the 6:40 intelligence brief, the 7:30 morning sweep, watchers across 30 job boards, a database sync on a four-hour cycle, an hourly on-site backup and a weekly encrypted offsite one, and the health checks that watch all of the above. Twice a day a heartbeat verifies that each scheduled output actually shipped, and when one did not, it prints an error ticket on a receipt printer beside my desk. At 6:25 each morning, before the heaviest hour of the schedule, a supervisor process clears the failure patterns I have seen before.

How the reliability layer works

The heartbeat trusts artifacts over exit codes because both failure directions are real: a job can exit 0 and ship nothing, and a job can exit dirty after shipping everything. So the check reads the output itself: did today's note get today's block, is the database fresh, did the backup snapshot advance.

Every change to the fleet has to pass a smoke test that runs inside the scheduler's own context, because a job that works in an interactive terminal can still fail under launchd, where permissions differ. Long-running agents sit under an escalating watchdog: a stall detector samples the process before killing it, so every kill leaves a diagnosis behind. The Claude CLI itself is pinned to a known-good version, and a gate job advances that pin only after the new version passes a live multi-turn test, twice.

The mini is the only machine that matters. The laptop and phone reach it over a private mesh network, so the same system answers wherever I am.

The incident log

What broke

Three failures since June were serious enough to get written post-mortems. One of the guards they exposed is still unbuilt. These tickets are the honest version of the section above.

Jun 23 · every overnight job killed

What happened: the whole overnight fleet died inside a 90-minute window, silently.

Root cause: operator hygiene. Ten terminal sessions I had abandoned over the prior week were holding about 1.5 GB of swap, and macOS killed background jobs to relieve the pressure.

Fix: a lock so only one heavy agent runs at a time, and a 4 a.m. reaper that closes idle sessions before they pile up.

Jun 25 · bot alive, not listening

What happened: the Slack bot showed a healthy process for 36 hours while its connection sat wedged in a silent reconnect loop.

Root cause: the process never died, so the operating system's auto-restart never fired.

Fix: the bot now exits on its own after six consecutive socket errors, which forces a restart, and the heartbeat gained a separate check for recent bot activity.

Jul 6 · 19 hours without a backup

What happened: an interrupted backup at 4:18 in the afternoon broke the hourly schedule, and every retry failed quietly for 19 hours while the interface claimed the next backup was coming. The 32-job morning in the header ran hours before this started; the check that would have caught it did not exist yet.

Root cause: the backup destination never remounted after the interruption. One manual mount fixed it.

Fix: the heartbeat now watches backup freshness directly. The guard that would remount the disk automatically is designed and still unbuilt. It is on the list.

Memory

One database under everything

Everything the system and I do ends up written down. A plain-text vault of 5,759 notes holds the projects, people, decisions, and daily journal. Under it sits a local database that mirrors the rest of my working life: 65,424 semantic embeddings, 29,169 iMessages, 518 logged working sessions, and a Gmail archive of 323,753 messages reaching back to 2009. Recall works by meaning, so a vague question still finds the right note, and the embedding models that power it run on the machine itself. The database never syncs anywhere, and the offsite backup copy is encrypted before it leaves.

How recall works

Three layers. The vault is the working set: plain markdown, readable in any editor, organized consistently enough that agents navigate it without special tooling. The lake is the archive: messages, mail, highlights, and session history stitched into one local SQLite database. The semantic layer indexes both with embeddings computed on the machine itself, using a small local model, so a question like "what was I working on last Tuesday" comes back with the journal entry, the messages, and the notes that touched it.

The privacy boundary is deliberate rather than absolute. Agents send what they retrieve to the Claude API while they work; that is the tool doing its job. What stays local is the archive itself: indexing runs on local models, the database syncs nowhere, and the weekly offsite copy is encrypted before upload.

Knowledge pipelines

Every card cites its page

I distill books into knowledge bases where every card cites the page it came from, and a card that fails citation gets deleted. The formal gate has run once: on June 28, all 1,889 cards from the first 10 books passed a full citation check. The corpus has since grown to 3,335 cards across 23 books through the same pipeline — each batch citation-checked at extraction, with the full-corpus formal gate run once so far, at 1,889. Two smaller banks sit beside it, built the same way: 619 cards of supply-chain signal and 379 cards of consulting craft.

How the pipeline verifies itself

Extraction runs as a fan-out of about thirty agents per batch, with a separate three-agent verification pass on a stronger model behind it. The two passes are kept apart on purpose, so the reviewer never grades its own work. One large batch ran 45 blocks and 3.65 million tokens end to end.

The gate is strict because these cards are built for paid advisory work, where an uncited claim is a liability I would own personally.

Darkroom Developments

Shipped in public

Latitude × Longitude is a pair of reinforcement-learning environments that simulate GPU procurement through a supply crisis, with scenarios corroborated against real constraint data. I built them solo and entered them at a hackathon run by HUD, a Y Combinator-backed startup, on June 20. One graded rollout hit the environment's maximum reward of 1.000. That is a single rollout.

What the environments actually test

An agent has to buy and allocate scarce GPU capacity while a crisis moves under it, with allocations tightening and spot prices spiking mid-run. Every scenario is deterministic, so a run can be replayed and graded, and the reward is normalized against optima found by brute force rather than tuned by hand. The result I found most interesting was qualitative: in one rollout the model surfaced a three-chip sourcing strategy the scenario never suggests. I have not published that trace.

What the 1.000 does not mean: it is one graded rollout on one scenario, not an averaged benchmark, and I have not published learning curves.
